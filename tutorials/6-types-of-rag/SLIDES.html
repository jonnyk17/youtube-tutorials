RAG 6 Types With Code

6 Types of RAG,
Clearly Explained

RAG is not one technique. It's a spectrum from simple file loading to agentic retrieval. Most developers default to vector search. There are five other options they're ignoring.

The Core Idea

RAG is just: fetch information, give it to the LLM.

How you fetch that information is the whole game. Everything else stays the same.

def rag_response(user_query: str) -> str: context = retrieve(user_query) # THIS is the only part that changes response = client.responses.create( model="gpt-5.4", input=f"Context:\n{context}\n\nQuestion: {user_query}" ) return response.output_text

That retrieve() function. That one line. That's what this entire video is about. Six ways to implement it.

Run the examples

Clone the repo. Run each script. The best way to understand this is to see it work. It clicks after a few runs, not from watching once.

Focus on the judgment

AI can write the code for you. But it can't choose which pattern to use. That's the skill this video teaches.

Don't try to memorize the code. Focus on understanding when and why you'd pick each approach. That's what separates a demo from something you can put in front of real users.

Type 1

Document Loading

Load entire files into context. No database, no embeddings, no chunking.

When to use

Small document sets. FAQs, policies, runbooks, recipes. Anything where partial retrieval loses meaning.

Why it works

No lost context. The LLM sees the entire document. More accurate for step-by-step content where order matters.

Where it breaks

Doesn't scale to many documents. Uses more tokens. Not suitable when you need to search across thousands of files.

def retrieve(query: str) -> str: docs = [] for path in Path("docs/").glob("*.md"): docs.append(path.read_text()) return "\n\n".join(docs) # That's it. No database. No embeddings. # Just files in a prompt.
Type 2

Full-Text Search

Keyword matching. When exact terms matter more than meaning.

When to use

User knows exactly what they want. Brand names, product codes, specific terms. "Nike" means Nike, not Adidas.

Built into Postgres

tsvector and tsquery. No extra infrastructure. Already in your database.

Where it breaks

"Comfortable shoes for long distance running." It looks for "comfortable" literally. Misses "cushioned," "supportive," "designed for endurance."

SELECT name, description FROM products WHERE to_tsvector('english', description) @@ websearch_to_tsquery('english', $query) ORDER BY ts_rank(...) DESC LIMIT 5; -- "blue running shoes" ✓ -- "comfortable for distance" ✗
Type 3

Vector Search

Semantic similarity. Understands what you mean, not just what you said.

When to use

Natural language queries against unstructured text. Product descriptions, reviews, documentation. "Comfortable for distance" finds "cushioned midsole."

Same database

pgvector in Postgres. No separate vector database needed. Just a different column type.

Where it breaks

"Nike shoes under $100." Vector search doesn't understand price as a filter. Returns $200 shoes because the description is semantically similar.

SELECT name, description, 1 - (embedding <=> $query_vec) AS similarity FROM products ORDER BY embedding <=> $query_vec LIMIT 5; -- "comfortable for distance" ✓ -- "Nike under $100" ✗
Type 4

Hybrid Search

Best of both worlds. Keyword precision + semantic understanding, merged with Reciprocal Rank Fusion.

The production default

If you're not sure which approach to use, start here. Runs both keyword and vector search, merges results with RRF scoring.

One Postgres query

"Nike running shoes comfortable for distance." Keyword locks in "Nike." Vector finds comfort descriptions. Combined: Nike shoes that are actually comfortable.

Where it breaks

"Shoes under $100 with 4+ stars." Price and ratings are structured data. Search isn't the right tool for filtering on exact values.

WITH fts AS ( SELECT id, ts_rank(...) AS rank FROM products WHERE tsv @@ query ), vec AS ( SELECT id, ROW_NUMBER() OVER(...) AS rank FROM products ORDER BY embedding <=> $vec ) SELECT id, 1.0/(60+fts.rank) + 1.0/(60+vec.rank) AS rrf FROM fts JOIN vec USING(id) ORDER BY rrf DESC;
Type 5

SQL RAG

The one nobody talks about. When your data is structured, generate queries instead of searching text.

5a: Parameterized (the safe default)

The LLM extracts filters from natural language. Those parameters slot into a pre-written query you control. Safe, fast, predictable.

5b: Dynamic (for flexibility)

The LLM writes the full SQL itself. Needed for GROUP BY, aggregations, and questions you can't predict. More powerful, more risk.

Where it breaks

"Something good for trail running in wet conditions." There's no "wetness" column. You need semantic search for that.

-- 5a: LLM fills in the form, you write the query -- User: "Nike running shoes under $100" -- LLM extracts: {brand: Nike, category: running, max_price: 100} SELECT name, price, rating FROM products WHERE brand = %s AND category = %s AND price <= %s
-- 5b: LLM writes the SQL from scratch -- User: "Which brand has the highest average rating?" SELECT brand, AVG(rating) FROM products GROUP BY brand ORDER BY AVG(rating) DESC LIMIT 1;
Type 6

Agentic RAG

The agent picks the right retrieval strategy for each part of the question.

Example query

"I want running shoes under $150 but what's your return policy if they don't fit?"

Tool call 1

Product Filter

"Running shoes under $150" is a standard lookup. Extracts parameters, runs a safe query.

Tool call 2

Document Loading

"Return policy" is a small policy doc. Loads the whole file. No search needed.

The agent has five tools: document loading, keyword search, vector search, product filter, and dynamic SQL. It picks the right one for each part of the question. This is where the spectrum pays off.

The Takeaway

Start simple. Move right only when simple breaks.

Don't over-engineer it. If your docs fit in context, just load them. Postgres handles full-text, vector, and SQL in one database. You don't need five different tools.
