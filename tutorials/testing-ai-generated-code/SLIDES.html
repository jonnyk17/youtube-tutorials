Testing TDD AI Agents

How I Test
AI-Generated Code

AI agents write code that looks correct and breaks in ways you don't expect. Tests give the agent a closed feedback loop. Write code, run the tests, see what failed, fix it.

Why Test?

Two reasons. Both matter more with AI agents.

Reason 1

Correctness

The code does what it's meant to do right now. The scoring function returns the right score. The login rejects the wrong password. The API returns the right data.

Reason 2

Regressions

Changes to one part of the codebase don't break something else. The agent fixes the search endpoint and accidentally breaks authentication. Tests catch that.

AI agents make dozens of changes in a single session. They're focused on the task in front of them. They have no idea they just broke something else.

The Pyramid

Four layers. Each catches something different.

MOST TESTS HERE

Unit Tests

Single functions. Fast. Cheap. Hundreds of them.

FEWER

Integration

Components together. API + database. Services connected.

FEW

E2E Tests

Full user flows. Real browser. Slow but tests what users see.

NEW LAYER

LLM Evals

Non-deterministic output. AI judges AI. Fuzzy but necessary.

Most of your testing with AI agents happens at the unit and integration level. That's where the feedback loop is tightest.

What to Test

Test your logic. Skip everyone else's.

Business logic

Scoring, pricing, access rules. If this breaks, it costs money or trust.

Complex conditionals

Multiple branches, edge cases. If you can't hold all paths in your head, write a test.

Data transformations

Input shape to output shape. Easy to get wrong, easy to test.

AI failure modes

Null handling, boundaries, error recovery. AI nails the happy path, misses everything else.

Framework boilerplate

Testing that FastAPI returns 200. You're testing their code, not yours.

Library behavior

Asserting json.loads parses JSON. The standard library works.

Mocks testing mocks

Asserting a mock was called. When the real thing breaks, the test still passes.

Simple CRUD

No custom logic? The framework handles it. A test here tests the framework.

The Problem

When AI tests its own code.

What happens

The agent grades its own homework

You say "write tests for this code." The agent reads the code and writes tests that confirm what it already did. Every test passes. You feel safe. But the tests prove nothing.

The real danger

False confidence

You can have a green test suite full of tests that test nothing useful. Framework boilerplate, library functions, implementation details. The feeling of safety without the reality.

More tests isn't better. Better tests are better. The prompt shapes the quality of the tests more than the AI does.

The Approach

Test-first vs test-after.

Test-After

Write code, then test

Better than no tests. But with AI agents, the agent reads its own code and writes tests that confirm what it already did.

It's grading its own homework.

Test-First (TDD)

Write the test, then the code

The test defines what the code should do before it exists. The agent has a concrete target. It writes the minimum to pass. No over-engineering.

You define "done." The agent gets there.

Test-first works with agents because it constrains them. The agent writes the minimum code to pass the test. Without that constraint, agents over-engineer everything.

The Cycle

Red, green, refactor.

# Your entire TDD instruction to the agent: "Use red-green TDD." # Five tokens. That's it. # Every good coding agent knows what this means.
The Workflow

Tell the agent what could go wrong.

Wrong

Test the functions

Write unit tests for the auth service. Cover password hashing, session tokens, and session expiry. # Produces 7 trivial tests. # Tests bcrypt. Tests secrets. # Tests nothing useful.
Right

Test what could go wrong

/tdd Think about what could actually go wrong. Don't test library functions. Test our decisions and security risks. # Produces 6 meaningful tests. # Every one proves a real requirement.

The prompt shapes the quality of the tests more than the AI does. You decide what to test. The /tdd skill handles the red-green cycle.

Examples

Good test vs bad test.

GOOD: TESTS BUSINESS LOGIC
def test_zero_experience_perfect_skills(): candidate = Candidate( years_experience=0, skills=["python", "fastapi"] ) job = Job( required_skills=["python", "fastapi"] ) score = calculate_score(candidate, job) assert score > 0.5 # Tests a real edge case. # Defines expected behavior. # Catches real bugs.
BAD: TESTS THE FRAMEWORK
def test_health_returns_200(): response = client.get("/health") assert response.status_code == 200 def test_sends_email(mock_smtp): send_welcome_email("user@test.com") mock_smtp.send.assert_called_once() # Tests FastAPI, not your code. # Tests a mock, not email delivery. # False confidence.
The Takeaway

More tests isn't better.
Better tests are better.

Tests are free to write now. But bad tests are as expensive to maintain as they've always been. Your job is to decide what matters. That judgment is the skill.
