01 / Requirements & Planning

What are we building?

Think before you code. Then use AI to think faster.

Real example
On a recent freelance project I spent 3 days on research, planning and design before writing a single line of code. The project went smoothly because of it.
PRD — Product Requirements Doc
  • →What are we building?
  • →Why are we building it?
  • →Who is it for?
  • →In scope / out of scope
Research phase
Using an external API or unfamiliar library? Have Claude research it first and cache the output into a research.md file. Prevents wrong turns mid-build.
Prototyping as planning
Build 2–3 throwaway versions to figure out what you're building. AI makes building cheap enough to use as a thinking tool. This is still planning — you're figuring out what to build.
PRD Spec research.md Context files RALF research phase Throwaway prototype

Great use of AI here

03 / Breaking Down The Work

Break it into bounded tasks

One task. One agent. Reviewable increments.

The principle
Don't hand an agent your entire app and say "build this." Give it one specific, bounded task with all the context it needs. The difference in output quality is dramatic.
Task breakdown example
01Add email validation to signup form
02Create GET /users/:id endpoint
03Add rate limiting to auth routes
04Write tests for auth flow
Tool options
  • →Linear — proper project management
  • →GitHub Issues — kanban built in
  • →Notion — flexible, good for solo
  • →A plain doc — works fine

The tool doesn't matter. The discipline of breaking things down does.

AI helps here too
Describe what you're building. Ask Claude to break it into tasks. It's surprisingly good at this — use it.
Kanban Linear GitHub Issues Parallel agents Task decomposition

04 / Build

Brief the agent. Don't just prompt.

Context is everything.

Without context
"Build me an auth system" → agent guesses → you get something that almost fits → hours spent fixing misaligned assumptions.
With context
An LLM makes hundreds of small decisions as it writes. Context means good decisions. No context means guessing. The output quality difference is not small.
What a good brief includes
  • 01Background on what you're building
  • 02Your technical design decisions
  • 03The specific task clearly stated
  • 04Patterns and conventions to follow
  • 05Things to avoid
Claude Code Cursor Kiro OpenCode Agentic loop YOLO mode Context window

05 / Review

The most skipped step

Never accept the first pass.

01
Build
Agent completes the task
02
Self-review
Ask the LLM to review its own code. Look for bugs, edge cases, security issues. It almost always finds something.
03
Human review
Understand what was built. Does it match the technical design?
04
Next task
Repeat the cycle
Why self-review works
Generation and review are different cognitive tasks. The LLM catches things on review it missed while building. Consistently. Almost every single time.
Common catches
  • →Edge cases & error handling
  • →Security vulnerabilities
  • →Missing input validation
  • →Performance issues
AI code review PR review Static analysis GitHub Actions QA plan
