The Problem

Code review was already the bottleneck.

I've seen developers take hours or days to review a single PR. Most teams struggle to review two or three a day. It was never fast enough.

2-3
PRs reviewed per dev per day
Before AI. Already a bottleneck.
10x
More PRs generated with AI agents
Same number of developers reviewing.

The problem isn't that AI code is bad. It's that we're generating so much of it that our traditional review process can't keep up. We need to change how we work.

01

AI Review vs Human Review

Complementary, not competing. The goal: make human review time count.

AI Review
Pattern matching, speed, consistency
  • Security scans (OWASP, injection, secrets)
  • Bug detection (logic, null, races)
  • Style guide adherence
  • Scales to any volume
  • Never gets tired
Handles mechanical checks at machine speed
Human Review
Judgement, context, strategy
  • Does this solve the business problem?
  • Is this the right architecture?
  • Does it meet acceptance criteria?
  • Will this cause problems in 3 months?
  • Should we even build this?
5 min focused > 30 min line-by-line

The 4 Layers

Each layer filters problems so the next sees less noise.

Layer 1
Automate the Obvious
Linting, tests, security scans. Via hooks. No judgement needed.
Layer 2
Review Locally
Run the code. Read the diff. Get an AI review in fresh context.
Layer 3
Review on CI
Automatic on every PR. Safety net for what you forgot.
Layer 4
Human Review
Architecture, business logic, judgement. Everything else is handled.

Layer 1

Automate the Obvious

Don't make people review things automated tools could catch.

Without hooks
  1. Claude writes code
  2. You review, notice style issues
  3. Push to CI, security fails
  4. Fix, push, review again
Multiple round trips
With hooks
  1. Claude writes code
  2. Hooks enforce checks automatically
  3. You review clean code
  4. CI passes first time
One pass

The same checks that run in CI now run locally before Claude can finish. Reviewers focus on logic and design instead of catching lint errors.

Layer 2

Review Locally

Before you push. Run it, read it, get a second opinion.

Step 1

Actually run the code

Click through the feature. Make sure it starts. I've seen PRs that were obviously broken. Don't be lazy.

Step 2

Read the diff

What files were touched? Does the scope match? AI's #1 issue: silent side effects.

Layer 4

Human Review

Three layers of checks have already run. The human focuses entirely on judgement.

Already handled What the human reviews
Lint errors, formatting, type errors Is this the right architecture?
Failing tests, broken builds Does this solve the business problem?
Known security issues (OWASP, injection) Does it meet acceptance criteria?
Obvious bugs, over-engineering Will this cause problems in 3 months?
Style violations, unused code Should we even build this?

5 minutes of focused human review on architecture and business logic is worth more than 30 minutes of line-by-line reading.

The Takeaway

There is no perfect tool.
The value is in layering them.

Start with Layer 1. Add layers as you need them. Every issue you catch becomes a rule. The system gets better over time.
